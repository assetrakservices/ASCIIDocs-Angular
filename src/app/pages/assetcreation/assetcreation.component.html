Asset Creation

1) Types of Assets

Following types of assets can be created at this stage:

  • Capitalized (Default)

  • Non-Capitalized (Cannot be retired. Needs to be mapped with capitalized asset.)

  • Expensed out

  • Leased

  • Two more for future

2) Capitalized Assets

a) Asset Creation

  1. Capitalized assets can be created through:

    • Creation form (Generally by those companies who are using AssetCues as the primary source for maintaining asset data).

    • Upload.

    • Integration with ERP/FTP (This will be used by companies who are using ERP/other accounting software for asset creation).

  2. If the asset is created through integration/multiple uploads, it could also have assets that are already created and need to be edited. Call "edit during upload" field update configuration for such assets. Remember that "edit during upload" will be a different configuration than the Editable information configuration. (Refer to the edit assets note)

  3. Show the history of all uploads. Store creation date/created by, etc., for each Asset, whether they are created through a form or bulk upload. In the case of integration with SAP/FTP, we need to create one integration user, which will be a default user with rights of inbound and outbound as per the configuration. This user’s name will get updated in the "created by" field.

  4. Special characters need to be dealt with properly. This has been a challenge in our current application.

  5. In single asset creation, documents can also be uploaded at the time of creation itself. For assets created through bulk upload and integration, documents can be uploaded through edit assets (Edit assets option). Following categories and sub-categories of documents will be there:

(1) Finance (Sub-category - Invoice, GRN, PO, other, etc.) (2) Contract Documents (Lease, Insurance, Warranty, AMC, etc.) (3) Asset Manuals (Maintenance manuals, usage manuals, etc.) (4) Transfer documents (Uploaded at the time of transfers) (5) Retirement document

Under each category above, multiple documents can be uploaded by selecting the sub-category of the document. In case another sub-category is selected, ask for the name of the document. Every category of document should be associated with only one view and can be accessed according to the view access. The association will be managed in the backend by us. A new document category can be created and assigned to a view by the company user.

  1. Approval workflow can be associated with create asset functionality only for assets capitalized in AssetCues. In that case, assets will be held in the review stage until approved. (Refer to the existing functionality for the same).

  2. At the create asset stage, we will provide 3 types of fields that can be managed through views:

(1) Some standard fields will be mandatorily required in the application and cannot be changed by the user. Examples would be location, asset class, cost of the asset, etc. (2) Some fields will be available through views for the user to use and enable for the company. They can disable or mark these fields as mandatory/non-mandatory or editable/non-editable in create asset and editable during upload configuration. (3) Users can create their fields in the views related to create asset and editable during upload and mark them as mandatory/non-mandatory and editable/non-editable at create/edit assets page.

  1. An email should be sent out after upload activity with the count of assets uploaded successfully and assets in error with a link to access the assets in error. In the case of integration with FTP/ERP, we need to send an email only if there are any errors or the program has not run.

  2. Refer to AE-10 and AE-708.

  3. Following fields will generally be used by the companies on the create asset page. - Refer to an excel sheet here.

(1) Asset Number An option to auto-generate the asset number in case of customers creating assets in AssetCues should be available. This should be available in configuration for all types of assets as to how they should be numbered (Logic).

(2) Asset Name (Description)

(3) Asset Acquisition Cost

(4) Put to use date

(5) Asset Class

(6) Count

(7) Asset Location

(8) Useful Life

(9) Acquisition Date

(10) NBV (If not provided, the cost will be NBV)

(11) NBV Date (If not provided, the acquisition date will be NBV Date. If cost and NBV are different, then the acquisition date and NBV date cannot be the same)

(12) Transaction Currency (Default will be the LE currency. If it is different, then it needs to be converted to the base currency)

(13) Other fields

(14) Retirement date can come in case of integrations (retire the asset if that date is available)

(15) Inventory number (in case, the customer has already tagged assets)

b Integrations/Bulk Upload

  1. In SAP Check if any asset has an old asset id; if yes, then check if that old asset is retired. If yes, then simply update asset details of the new asset according to the incoming file, including the barcoding details from the retired asset. For the current status, check if the asset transferred from one location to another, then barcoding details transfer or not. If not, then only do this change.

  2. We need to develop a flow of assets being edited in bulk (through upload of assets but the editing will happen through edit API). If the location or retirement indicator or asset class is changing in ERP, we need to keep a check on stages and status when this change can be done. Otherwise, hold back on such changes if the asset is under audit, transfer, or retirement in AC.

  3. In case of integrations, we need to send an email to the customer informing any failed upload (scheduler could not run), any asset in error (mandatory information missing, validations failed, etc.), or assets held for update (as that asset is locked for editing) so that they are aware some action is required from their side.

c) Edit During Upload

  1. If the asset number and sub-number are the same in single asset creation, we will throw an error and not allow to create. However, in case of bulk upload or integration, if an asset with the same number and sub-number is uploaded, we will edit the information as per edit during upload configuration.

  2. In case of split (quantity or components), Cost and NBV from ERP/upload file will be distributed to the individual components/quantities in the proportion of their cost and NBV to the total cost and NBV. If any component or quantity is retired in AssetCues, do not assign value to that component/quantity. (Partial retirement case).

d) Validations/Handling Errors

  1. There will be 5 types of validations:

(1) Mandatory information check. Whatever is marked as mandatory at the time of creation should be available in the asset record proposed to be created. The mandatory/non-mandatory configuration will be managed through views.

(2) Dependent field check: (a) NBV cannot be higher than cost. (b) Sub-location and location should be mapped. (c) Acquisition date and put to use date should be less than the current date. (d) NBV date should not be higher than acquisition date. (e) Cost and NBV should be greater than zero.

(3) Data type Validations.

(4) Duplicate errors - duplicate asset and sub-number combination. (only in case of single asset creation. In case of bulk upload, this will be managed through edit during upload functionality). Serial number also cannot be duplicate.

(5) Master check (Location, class, sub-location, etc. are there in our master or not).

  1. If the validations fail, the data should be held in the "Errors page".

  2. Provide action buttons in front of the assets in error where if the location was missing, give an option to create a location, etc., if the user has authorizations. If the error is of missing data or mapping, this can also be updated by the user from this page itself.

  3. We need to give an option to manually run the error validations again to resolve an error on any asset if the error in those assets is resolved. (by resolving it from a different page or on a different asset) Eg. an asset was in error because the asset class was not available in the asset master. Now, if that asset class is added and I run the validations again on the error page, it should run the error validations again and remove this record.

  4. Give an option to delete the asset at the error and review stage. Only till an asset is successfully created can an asset be deleted. At all other stages, it can only be retired.

e) Fields/Views

  1. There will be a different configuration of the mandatory and editable fields for different types of assets such as capitalized assets, non-capitalized assets, expensed out or leased assets.

f) Logic to be Executed when Creation is Successful

  1. If an asset is uploaded with an inventory number, the stage to be updated in this asset will be different from the asset uploaded without an inventory number. (check the stages file). These assets shall not go to the pre-tagging but directly to the available for inventory stage.

  2. Update expiry date based on the useful life.

  3. Retire the asset if the retirement date is there in the uploaded assets. Change the stage too.

  4. Update default values if certain values are blank such as qty, criticality, sub-no., condition, NBV, depreciation date, etc.

  5. Date format as per the configuration.

g) Mapping

  1. Whenever any field is mapped in Excel is mapped with a field of AssetCues by the user, we should store that mapping in the database. Whenever we find the same field in future spreadsheets being uploaded, if we have that column mapped with one of our fields, we should show that as mapped in the dialogue. We will not check the user who has uploaded the data (We are currently saving this mapping user-wise) - Instead, it would be field-wise. One field of AssetCues can be mapped with multiple fields as different users may upload different templates. However, one field in Excel can only be mapped with one AssetCues field (if this mapping is changed, we need to overwrite the earlier mapping).

  2. In case of integration, the mapping should be stored in advance. However, if it is not shared and integration scheduled, then there could be a problem. We need to discuss this journey.

  3. In case of asset creation through integration, the mapping of asset in ERP should always be stored with the original asset in AssetCues so that if an asset is split, it does not affect the mapping.

  4. Mapping of fields should be stored at the group level.

h) User-Related Fields

We should not allow assets to be uploaded with user-related fields (user-related field should not be configured in asset creation view). However, we need to give an option to bulk update user details for multiple assets in the assignment module (Link assignment module).

i) Configuration

In the configuration, ask whether the assets will be capitalized (only applicable for capitalized assets) in AssetCues or External source (FTP or ERP). If it is configured to be created in AssetCues, ask whether the asset number should be auto-assigned or manually created. If the source is external, ask whether the user will upload or use integrations. In the case of upload, keep the upload button enabled and disable the form. If it is integration, ask integration details and disable the form as well as upload option. The form and upload in either case will be disabled only for the creation of capitalized assets. For other asset types, it will still remain enabled. Approval workflow can only be configured for assets created in AssetCues. For creation through integration, workflow can be configured.

j) Fields for Create Asset View

Only for the create asset view, we will need fields for upload, mandatory during upload, and editable during upload for all asset types.

3) Non-Capitalized Assets

a) Non-capitalized assets can only be created through a creation form or bulk upload. b) Option for the creation of non-capitalized assets should be enabled from configurations (It should be disabled by default). If enabled, ask for the numbering logic (Link this with configuration).

c) Following fields should be mandatory:

  1. GRN Number (Sub number in case of multiple entries with |1 and so on can be made as 1 GRN can have more than one items)

  2. Asset number (Non-mandatory. In case the user knows under which ID the capitalization would happen when the invoice is received. This will help map the capitalized and non-capitalized assets in the future for common numbers)

  3. Asset Name (Description)

  4. Asset Acquisition Cost

  5. Asset Class

  6. Asset Location

  7. Count

  8. GRN Date

  9. Transaction Currency (Default will be the LE currency. If it is different, then needs to be converted to the base currency)

  10. Other fields

d) Non-capitalized assets will be treated the same as capitalized assets in the system, except that non-capitalized assets cannot be retired in the application.

e) An option to map non-capitalized assets with capitalized should be available:

  1. Non-capitalized asset should not be under pending audit or pending transfer at the time of mapping.

  2. Once mapped, the non-capitalized asset will be disabled and unavailable. Should not be shown on any pages except the history reports (Tagging, transfer, assignment, and audit trail).

  3. Ask the user what information of non-capitalized assets needs to be mapped with capitalized assets. It could be user information (assignment), Maintenance information, etc. Wherever information is available in both the assets being mapped (capitalized and non-capitalized), ask the user which information needs to be updated.

  4. The mapping should be an event and should be available in the asset trail report of the asset. One item should be shown mentioning that a non-capitalized asset has been mapped with a capitalized asset and accordingly, some information of the capitalized asset got updated. The history will remain under non-capitalized assets but a link to go to that history from capitalized asset history should be available.

f) If a non-capitalized asset is not mapped with capitalized assets for a period of "configurable days," there should be notifications regarding the same.

g) These assets should not form part of the fixed assets register by default. Create a separate report for these assets.

4) Expensed Out Assets

a) These are those assets that are charged off in the profit and loss and not capitalized by the company but they still need to track these assets.

  1. Invoice Number

  2. Asset Name (Description)

  3. Asset Category

  4. Type

  5. Sub-Type

  6. Asset Location

  7. Count

  8. Acquisition Date

  9. A cost field separate from the asset cost should be created

  10. Other fields

b) Expensed out assets are the same as capitalized assets, just that there will be a filter on all pages to exclude these assets, mainly in the fixed asset register the default report would exclude expensed out assets.

c) These should be automatically numbered in the application using the logic set out in the configuration.

5) Leased Assets

a) These are those assets which are leased from a vendor and hence are not capitalized by the company but they still need to track these assets.

(1) Asset Name (Description) (2) Asset Category (3) Asset Type (4) Asset Sub-Type (5) Asset Location (6) Count (7) Acquisition Date (8) Other fields (9) Lessor Name (10) Lease end date (Similar to asset expiration date) However, after this date, the alerts should be sent out for assignment removal.

b) Leased assets are the same as capitalized assets, just that there will be a filter on all pages to exclude these assets, mainly in the fixed asset register the default report would exclude leased assets.

c) These should be automatically numbered in the application using the logic set out in the configuration.

d) There should be a process to hand over leased assets to asset owner.
